**Why prior sessions never reached `complete`** (all stuck at `in_flight`): the out-of-process `ExternalWatchdog` SIGKILLs (exit 137, reason=heartbeat_stale) when the loop heartbeat (written on-loop by WallClockWatchdog tick, 120s stale threshold via `JARVIS_EXTERNAL_WATCHDOG_STALE_S`) freezes. Four distinct wedges, fixed in order (each unmasked the next):
1. **Oracle over-indexed `.claude/worktrees`** — `should_exclude()` substring-matches EXCLUDE_PATTERNS; `.worktrees` never matches `.claude/worktrees` (slash≠dot), so it ast-parsed 29,558 duplicate .py (6× the 6,051 real tree) every cycle → process-pool saturation → 73s loop block. Fix: add `.claude` to Oracle EXCLUDE_PATTERNS + miner `_WALK_PRUNE_SEGMENTS`, + `_is_linked_git_worktree()` (skip any dir whose `.git` is a FILE = `git worktree add`; embedded clones like `backend/vision` keep their dir-`.git` and stay indexed — 229 real .py preserved).
2. **posture `commit_ratios` forked git ON the loop thread** — `commit_ratios_async` used `asyncio.create_subprocess_exec`, which forks/execs the child on the event-loop thread. From the huge multi-threaded organism process (concurrent with Oracle cold-index forking 16 workers) the fork blocked the loop 33–108s (git log itself is 0.14s — the cost is the FORK, not the query; varies with memory/thread pressure). Synchronous+yield-less, so the 30s collector `wait_for` can't cancel it and ControlPlaneStarvation stays SILENT (single sync wedge logs nothing). 108.9s came within 11s of SIGKILL. Fix: run the git calls in the dedicated `fs_signal_executor` (the pool other fs-signals already use) so a slow fork blocks a worker thread, never the loop. Slice 52 HEAD-cache + subprocess.run timeouts preserved. After: ControlPlaneStarvation 0, heartbeat fresh, no SIGKILL — the LoopSink still LABELS ~92s but loop is healthy (it measures async-block wall-time, not loop-block).
3. **shutdown wedged on Oracle cache write** — `_shutdown_components` → Oracle `_write_cache_blocking` (`cache_path.write_bytes` of the graph cache) can't be interrupted by its 5s deadline (blocking C write), exceeding the 30s BoundedShutdownWatchdog → `os._exit(75)` before `_generate_report` stamps complete. Fix (operational, designed flag): `JARVIS_ORACLE_SHUTDOWN_DEADLINE_S=0` skips the shutdown cache save (rebuilds next boot). [Deeper follow-up: make the cache write bounded/interruptible.]
4. **harvester false-positive OOM** — `telemetry_parse._RE_OOM` matched bare `process_memory_cap`/`OOM`, which also matched the ProcessMemoryWatchdog *arming* line ("armed: … stop_reason=process_memory_cap before OS OOM-kill"). A clean wall_clock_cap stop at rss=555MB was mis-verdicted ANOMALY/OOM. Fix: match the FIRE ("stopping: process_memory_cap") + MemoryError/emergency_brake/critical-pressure, + OR `summary.stop_reason=="process_memory_cap"`. See [[project_slice235_validation_sixth_layer]] (telemetry harvester lineage).
