**P1 — Oracle IPC daemonic-pool fix (kills the starvation).** VERIFY-FIRST: the "robust IPC layer" the brief asked to engineer ALREADY EXISTS — `backend/core/ouroboros/oracle_ipc.py` (Slice 112) is spawn-context **Pipe** IPC (`multiprocessing.get_context("spawn").Pipe`) with crash-respawn, NOT naive multiprocessing. The crash was a precise defect: the Oracle worker is spawned **`daemon=True`** (`oracle_ipc.py:186`), and a daemonic process CANNOT have children — so the isolated Oracle's inner `ast_compile_helper` `ProcessPoolExecutor` (via `oracle._index_file` → `_process_pool_analyze_for_oracle`, the `loop.run_in_executor(pool, ...)` at the line that logs `pool dispatch failed: ... daemonic processes are not allowed to have children`) crashed ×4789 → 8-13s event-loop starvation. **Fix (compose, NOT rewrite):** the Oracle is ALREADY off the main loop in its own subprocess → the nested pool is redundant. Added gated in-process analyze mode to `ast_compile_helper.py` (`ast_helper_inprocess_enabled()`, env `JARVIS_AST_HELPER_INPROCESS_ENABLED`, default-FALSE): when ON, `_process_pool_analyze_for_oracle` dispatches the worker via `asyncio.to_thread` instead of the spawn pool. `oracle_ipc._oracle_worker_main` does `os.environ.setdefault("JARVIS_AST_HELPER_INPROCESS_ENABLED","1")` before `TheOracle` builds. Default-FALSE → main process keeps the spawn pool byte-identical. 5 tests + oracle_ipc(7)/ast_compile_helper(23) regression green. (NOTE: ast_compile_helper pool tests need sandbox-OFF to run — ProcessPoolExecutor spawn is sandbox-blocked, "Operation not permitted".)
