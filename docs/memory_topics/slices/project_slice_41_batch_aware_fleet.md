Slice 41 — Batch-Aware Fleet Eligibility & Ledger-Driven Routing. **MERGED to main 2026-05-28** PR #63747 (merge `f23b510a0c`). Opus-reviewed READY TO MERGE. Builds on [[project_slice_40_gls_boot_wiring]] + [[project_slice_39_multi_surface_health]].

**The decisive pre-build probe (settled whether this was a fix or a workaround):** an end-to-end batch-generation probe (upload→create→await→retrieve) returned **16 real tokens in 18s** on `Qwen3.5-35B-A3B-FP8` — the SAME model whose streaming probe returns `done_before_content` (0 tokens). So the v34 degradation is **transport-specific (RT-streaming wire), NOT generation-capability**. The batch lane works. This reframed the whole v25→v34 0-APPLY story: the **streaming-based preflight emptied the fleet** (`active=0 → PreflightDaemon backoff / all-failed halt`) before the working batch lane was ever reached.

**Phase 1 — batch-aware preflight eligibility (`preflight_probe.py`):** new `PreflightVerdict.ACTIVE_BATCH_ONLY` (taxonomy 5→6, AST pin updated). A `done_before_content` streaming result, when the surface ledger shows `batch_storage` HEALTHY, is upgraded from DEGRADED_5XX → ACTIVE_BATCH_ONLY: counts INTO `active_count` (fleet not empty → no halt/backoff), increments `active_batch_only_count`, stamps `entitlement_marker="FORCE_BATCH"`, and is NOT reported as a sentinel failure. `_classify_outcome` stays PURE; ledger read once/cycle via `_batch_surface_healthy` (gated on surface-health master, fails CLOSED). New pure helper `_is_streaming_only_degradation` (True ONLY for done_before_content; timeouts/5xx/entitlement are NOT upgraded — they'd break batch too).

**Phase 2 — ledger-driven routing (`doubleword_provider.py`):** `_slice36_should_force_batch` gains `_slice41_ledger_force_batch()` — forces batch on standard/complex (Claude-disabled gate retained) when streaming degraded + batch healthy, EVEN IF the static `JARVIS_DW_FORCE_BATCH_STANDARD_COMPLEX` opt-in is off (don't fail closed). Fails closed on any error.

**Phase 3 — scheduler test fix (`test_dw_heavy_probe.py`, subagent `1945b238bd`):** the 2 pre-existing `test_scheduler_*` failures were a STUB SIGNATURE DRIFT — `_StubProber.probe()` missing the `parameter_count_b` param `HeavyProber.probe` gained; `run_cycle` passed it → swallowed TypeError → run_cycle returned None. **KEY review clarification (Opus):** `ACTIVE_BATCH_ONLY`/`FORCE_BATCH` marker is NOT consumed for dispatch SELECTION — its ONLY effect is preventing the spurious all-failed boot halt. Routing is independently RE-DERIVED from the LIVE ledger at dispatch (`_slice36_should_force_batch`), so masking risk is bounded (a genuinely-dead model fails downstream via normal cascade/demotion, no black hole) and it's self-correcting (if batch later degrades, `_batch_surface_healthy` flips False). The `entitlement_marker="FORCE_BATCH"` is effectively dead metadata (live > snapshot) — harmless. The two phases COMPOSE: preflight no longer empties the fleet → ops dispatch → routing forces batch → batch generates.

**Known ordering nuance:** Slice 40 boot sweep is wired AFTER preflight in `_build_components`, so preflight reads the PRIOR boot's ledger. Steady-state fine (ledger persists on disk); a truly cold first-ever boot (no ledger) → preflight fails closed → halts → next boot's sweep populates it → 2nd boot works (self-heals across 2 boots). Future refinement: run the sweep BEFORE preflight for fresh same-cycle ledger.
