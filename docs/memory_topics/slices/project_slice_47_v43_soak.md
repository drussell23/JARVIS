**Slice 47 (Predictive Runtime Orchestration) — runbook's core mechanism REJECTED on architecture grounds, 2026-05-30.** The runbook authorized a "Dynamic Budget Escalation Waiver": query the Orchestrator op-ledger from inside the wall-clock evaluation thread and, if a worker reached APPLY/VERIFY/AutoCommit, extend the `--max-wall-seconds` envelope by `JARVIS_WAIVER_INCREMENT_S`. **Rejected** (operator concurred — "absolute tier-one systems architecture"): coupling the out-of-band `WallClockWatchdog` resource-zero kill thread (`harness.py::_start_wall_clock_hard_deadline_thread`, reads only raw `time.monotonic()`/`time.time()`, severed from logging/loop/SIGTERM) to the inner op-ledger means a *wedged* VERIFY (pytest hang = the v41 67-min class) keeps the extend-condition true forever → watchdog deadlocks WITH the system instead of killing it. Same principle that retired `JARVIS_WALL_CLOCK_ESCALATION_SIGTERM_S/_EXIT_S` (`harness.py:5489`): a watchdog sharing a resource — signal path, logging lock, OR state-ledger — with the system it guards is not a watchdog. Also: runbook env vars `JARVIS_SOAK_WALL_CAP`/`JARVIS_WAIVER_INCREMENT_S`/`JARVIS_SOAK_COST_CEILING` and entrypoint `./ouroboros_run.py` **do not exist**; real knobs are `--max-wall-seconds`/`OUROBOROS_BATTLE_MAX_WALL_SECONDS` + `--cost-cap`, real entrypoint `scripts/ouroboros_battle_test.py`. **Chosen instead (Option 1): raise the static cap** (the decoupled fix already recommended in [[project_slice_46_monotonic_deadman]]). Only deliverable landed: a **Watchdog Isolation Invariant** doc block added to CLAUDE.md beside the `--max-wall-seconds` paragraph (NOT yet committed — main kept pristine; doc lives on branch `ouroboros/slice-47-predictive-runtime-orchestration`). No waiver code written.
