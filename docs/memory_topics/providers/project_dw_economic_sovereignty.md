**DW economic sovereignty arc (Slices 168-171, all MERGED to main 2026-06-08).** Born from two Seb/Meryem @ Doubleword emails + the operator's standing refusal to fund Claude ("DW is the funded primary; stop cascading to expensive Claude"). The throughline: DW kept *cascading to Claude* on its own failure modes, defeating vendor sovereignty and overspending. Each slice closes one cascade source. Builds on [[project_slice131_cost_sovereign]].

**DoubleWord provider quirks (verified, load-bearing — `doubleword_provider.py`):**
- **`reasoning_effort: "none"` is REJECTED by some models** (deepseek-v4-pro → DW *cancels the batch*, surfaced async so it looks like a transport failure, not a 400). We derive `none` for trivial/simple ops (`_COMPLEXITY_REASONING_EFFORT`), so a trivial op routed there got cancelled → cascade.
- **High `reasoning_effort` RUPTURES the SSE chunked stream** (`ClientPayloadError: TransferEncodingError`) — the dispatch *mislabels it `live_transport`*. `none/low/medium` stream cleanly. We clamp the streaming ceiling to `medium` via `_dw_max_reasoning_effort()` (`JARVIS_DW_MAX_REASONING_EFFORT`, default `medium`); explicit `JARVIS_DW_REASONING_EFFORT` override is NOT clamped.

**Slice 168 — per-MODEL reasoning_effort FLOOR.** `_dw_model_min_effort(model)` clamps the resolved effort UP to a model minimum (so we never send a rejected value). Env `JARVIS_DW_MODEL_MIN_EFFORT="substr:effort,..."` (generic substring match — no hardcoded model in the *algorithm*; default `deepseek-v4-pro:low`). `_clamp_up_to_min` is the symmetric partner to `_clamp_reasoning_effort`. Threaded `model=_effective_model` into `_reasoning_request_params` at all 3 request-build sites; floors even an explicit effort.

**Slice 169 — DYNAMIC capability resolver.** `dw_catalog_client.parse_supported_reasoning_efforts(raw)` + `catalog_min_reasoning_effort(model_id)` read per-model supported efforts from DW's `/v1/models` metadata (via the existing disk-cached catalog; tries top-level keys + a `capabilities` sub-dict since DW hasn't finalized the field). `_dw_model_min_effort` resolves DYNAMIC-first → static env-map fallback (168) → `none`. When DW exposes the field, the floor self-resolves with zero static map. Byte-identical to 168 until then.

**Slice 170 — THE cost-cascade fix (the big one).** Root cause in `_slice36_should_force_batch` (`doubleword_provider.py`): `if not _claude_unavailable: return False` — DW only failed over to its OWN batch transport when **Claude was DEAD**. So whenever Claude had credit, a DW *stream rupture* (transport-specific) cascaded straight to the expensive Claude — DW was only "primary" when Claude was broke. **Fix:** consult the *existing* `_slice41_ledger_force_batch()` signal (surface-health ledger: `DIRECT_STREAMING` degraded AND `BATCH_STORAGE` healthy) **BEFORE** the Claude-availability gate → force DW-batch even when Claude is available, so a rupture fails over WITHIN DW. Claude stays the fallback for genuine DW-*wide* outages (both surfaces degraded). `JARVIS_DW_INTRA_TRANSPORT_FAILOVER_ENABLED` default-TRUE, failure-path-only (fires only on a degraded stream). Surface-health is default-ON (Slice 40); `_note_dw_live_transport_degraded` populates the ledger on rupture. Updated `test_slice41_batch_aware_fleet` (the old `test_force_batch_requires_claude_disabled` encoded the leak as an invariant).

**Slice 171 — economic telemetry (make the savings visible).** `economic_telemetry.py`: thread-safe `EconomicTelemetry` singleton (intra_failovers count + estimated capital saved; 832ns/record, lock+increment, no I/O/GIL stall). **Attribution with zero re-eval:** `force_batch=True` while Claude is AVAILABLE ⟹ the Slice 170 reroute fired (legacy batch path REQUIRES Claude unavailable) = a cascade avoided. `doubleword_provider._record_intra_failover_telemetry(ctx, force_batch)` records at the consumption point (~line 1821). `discord_gateway.economic_telemetry_line()` renders a `💸 sovereignty` field on every `APPROVAL_REQUIRED` embed. Env `JARVIS_ECONOMIC_TELEMETRY_ENABLED` (default-ON), `JARVIS_ECONOMIC_REROUTE_SAVED_USD` (default 0.05 = Claude-vs-DW delta for a std/complex op).

**Slice 172 — DW failure-risk predictor (the PREDICTIVE cortex).** Slice 170 is REACTIVE (fails over once the stream is CONFIRMED degraded); 172 FORECASTS. `dw_failure_predictor.py`: pure-Python recency-weighted **Poisson interval estimator** (only `math`/`collections`/`threading` — NO torch/tf). Rupture events land in a bounded monotonic-timestamp ring (`record_rupture`, fed UNCONDITIONALLY at `candidate_generator._note_dw_live_transport_degraded` — the SAME stream the surface-health ledger ingests; flag gates routing not data collection, so the forecast is warm the instant routing is enabled). Each event in a lookback window contributes a half-life-decayed weight; rate `λ = Σweights/lookback` → `P(≥1 rupture in horizon) = 1 - exp(-λ·horizon)` ∈ [0,1]. `_slice36_should_force_batch` gains a PREDICTIVE branch AFTER the Slice 170 reactive one: forecast ≥ threshold → preempt to batch BEFORE the stream breaks. Opt-in `JARVIS_DW_PREDICTIVE_ROUTING_ENABLED` **default-FALSE** (§33.1 — acts on a prediction, not a confirmed failure); env-tunable `JARVIS_DW_RUPTURE_{HORIZON,LOOKBACK,HALFLIFE,RISK_THRESHOLD}_S` (defaults 300/600/120s, 0.7). `discord_gateway.rupture_risk_line()` renders a `🔮 forecast` field (live rupture risk %) on every gate embed. A preemptive route is still a `force_batch` → Slice 171 telemetry records the capital saved. Proven: cold 0.0 → 4-rupture cluster 0.865 (preempt) → decayed 0.061. **KNOWN BLINDSPOTS (not yet built):** (A) ✅ **CLOSED (Slice 173)** — predictive detour now requires `_dw_batch_lane_healthy()` (reuses `preflight_probe._batch_surface_healthy`, fails closed); risky-stream + degraded-batch → abort detour, stay RT → correct cascade to Claude; (B) ✅ **CLOSED (Slice 175)** — `DWFailurePredictor._rings` is per-model (`{model_key→deque}`, case-normalized, `""`=isolated unknown bucket) + per-model calibrators/persist files (`get_threshold_calibrator(model_id)`, `dw_threshold_calibration_<model>.json`); `record_rupture(now,model_id)` fed at `_note_dw_live_transport_degraded`, `_slice36_should_force_batch(context,model_id=_effective_model)` queries the target model's forecast, Discord shows `highest_risk_model()`; proven deepseek 0.918→BATCH while qwen 0.000→STREAMS; (C) ✅ **CLOSED (Slice 174)** — `ThresholdCalibrator` self-tunes the threshold (FP→raise, FN→lower, bounded 0.30-0.95, step 0.02), tracks Brier, persists to `.jarvis/dw_threshold_calibration.json` (no restart amnesia), gated `JARVIS_DW_CALIBRATION_ENABLED` default-FALSE; proven 0.50→0.80 under 6 FPs, 0.70→0.40 under 6 FNs.
