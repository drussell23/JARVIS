## On-demand was a 3-LAYER inert-config bug (PRs #69666→#69669→#69670, 2026-06-22) — load-bearing lesson
The "on-demand" intent was overridden at THREE layers; each fix exposed the next because I verified the PRINTED LABEL not the ACTUAL VM scheduling. The node got `compute.instances.preempted` by GCP `system` THREE times before it stuck. Layers: (1) `ignite_sovereign_cloud_node` `cfg.use_spot=True` → made a `--spot` flag, default False (#69666); (2) `gcp_vm_manager._build_instance` scheduling site hardcoded SPOT → made conditional (#69666); (3) a SECOND scheduling site (`start_soak_vm`'s "invincible node" path ~10606) ALSO hardcoded `preemptible=True/SPOT` → made conditional (#69669); (4) **THE root cause:** `start_soak_vm` line 4684 `self.config.use_spot = True` FORCE-overrode the config regardless of caller intent, downstream of BOTH scheduling fixes → removed it (#69670). Only after #69670 did `gcloud describe ... scheduling.provisioningModel` finally read `STANDARD / preemptible=False`. **LESSON: verify the EFFECT (`gcloud compute instances describe` → actual scheduling), NOT the intent (the ignition's printed 'ON-DEMAND' label). A config flag can be force-overridden at N layers between set and use; grep ALL assignments (`use_spot = True`), not just the read sites.** The Autopsy's `instance not found` + the GCP ops log (`compute.instances.preempted ... system`) were the only way to diagnose preemption vs a sentinel kill.
