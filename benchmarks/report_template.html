In Plain English

TTFB is like the time between pressing "play" on a streaming service and hearing the first note of a song. If it takes 4 seconds (Daniel), you think the app is broken. If it takes 0.3 seconds (Voice.ai), it feels instant. In voice assistants, this delay is the difference between natural conversation and awkward silence.

Technical Detail

  • Definition: TTFB (Time to First Byte) is the elapsed time from HTTP request sent to first audio chunk received. Measured with time.perf_counter() at nanosecond precision for maximum accuracy.
  • Streaming vs sync: Streaming protocols (HTTP Stream, WebSocket) start delivering audio while still generating the rest. The sync endpoint blocks until the complete audio is ready. Daniel's say -o file must synthesize the entire audio before writing — no streaming is possible.
  • Test coverage: We tested 3 sentence lengths (26, 92, 377 characters) to capture scaling behavior across realistic input ranges. 128 total measurements were collected across 2 independent benchmark runs.
  • HTTP Persistent architecture: The flat TTFB curve (292/350/361ms across short/medium/long) proves the model begins audio generation almost immediately regardless of input length — a sign of true streaming synthesis.
  • WebSocket overhead: Fresh WebSocket connections measured 831-858ms due to per-request TLS plus WebSocket handshake overhead (approximately 600ms). Persistent connections would eliminate this entirely.
  • HTTP Sync scaling: TTFB scales linearly with text length (801ms to 2,115ms to 5,890ms for short/medium/long), confirming it waits for full generation before returning any data.

What the Data Shows

  • Protocol choice matters more than model speed: The data proves that choosing the right protocol has a larger impact on perceived latency than any server-side optimization could achieve.

8. Total Generation Time

Total time from request initiation to complete audio receipt. This includes TTFB plus the time to transfer the full audio payload. For streaming protocols, playback can begin at TTFB while the remainder downloads.

{{CHART_TOTAL_TIME}}
Total generation time context

In Plain English

TTFB is when you hear the first word. Total time is when the last word finishes. It's like a faucet — TTFB is how fast water starts flowing when you turn the handle, total time is how long until the glass is full. For streaming TTS, the user starts "drinking" immediately at TTFB, so total time is invisible to the experience.

Technical Detail

  • Definition: Total time = elapsed time from request initiation to last audio byte received. For streaming protocols, total time is greater than TTFB because audio generates and transfers progressively. For sync protocols, total time equals TTFB because nothing is delivered until everything is ready.
  • Daniel's total: Daniel's total includes synthesis time (TTFB) plus afplay playback duration. Audio duration scales linearly with text length (approximately 3 seconds for short, 10 seconds for medium, 30 seconds for long).
  • Streaming advantage: With streaming protocols, users begin hearing audio at TTFB while the rest downloads in the background. The total time is invisible to the user experience because playback overlaps with transfer.
  • Infrastructure relevance: Total time matters for connection lifecycle management, concurrent request planning, bandwidth allocation, and credit consumption rate — even though users do not perceive it directly.
  • Generation throughput: Voice.ai's approximately 6.5 seconds total for 377 characters equals roughly 58 characters per second generation rate. This determines throughput for batch use cases like audiobooks or report narration.

What the Data Shows

  • Short text: Voice.ai HTTP approximately 680ms total versus Daniel approximately 4,400ms. The user experience gap is even larger than the numbers suggest because Voice.ai starts playing at 292ms while Daniel waits 3,152ms.
  • Long text: Voice.ai approximately 6,500ms total versus Daniel approximately 12,900ms. But the perceived difference is massive — Voice.ai audio starts at 361ms while Daniel waits 3,627ms before any sound.
  • HTTP Sync penalty: HTTP Sync total equals TTFB (5,890ms for long text) because it blocks until full generation completes. This is the worst of all worlds — maximum wait with no streaming benefit.
  • Scaling behavior: Total time scales with text length for all protocols, but TTFB only scales for HTTP Sync. This confirms streaming protocols begin audio delivery before text processing finishes.

Why This Matters for Voice.ai

  • Infrastructure planning: Total time data helps customers plan infrastructure: connection pool sizing, concurrent request limits, timeout configuration, and bandwidth budgets.
  • Response header opportunity: Consider adding an X-Estimated-Duration response header so clients can pre-allocate playback buffers and display progress indicators for long utterances.
  • Batch throughput: For long-form narration (audiobooks, reports), total time determines throughput. The 58 characters-per-second rate should be documented for customers planning batch workloads.

Key Takeaway

Thanks to streaming, Voice.ai's total time is invisible to users — they start hearing audio at 350ms while the rest generates in the background.

15.1   Network Physics: The Theoretical Latency Floor

Even with perfect server hardware and zero queuing, a physical minimum latency exists — set by the speed of light traveling through fiber-optic cable.

Speed of light in vacuum: c = 299,792 km/s Refractive index of fiber: n ≈ 1.47 Speed in fiber: v = c / n ≈ 200,000 km/s Round-trip propagation delay for distance d: t_prop = 2d / v Estimated US residential → Voice.ai CDN edge ≈ 2,000 km: t_prop = 2 × 2,000 / 200,000 = 0.020 s = 20 ms TTFB decomposition (persistent connection, median short sentence): TTFB_measured = 272 ms (median, HTTP Persistent short) − t_prop = 20 ms (network round-trip, physics floor) − t_http = 5 ms (HTTP headers, framing) ───────────────────────────────── t_inference = 247 ms (server-side neural TTS generation)
Key finding: 91% of the 272 ms TTFB is server-side neural inference. Even deploying Voice.ai on the same machine as JARVIS (0 ms network RTT) would reduce TTFB to only ~252 ms — a 7% gain at best. The inference engine is the ceiling to optimize, not the network.
How to understand the speed-of-light analysis

In Plain English

Imagine ordering a pizza. Even if the delivery driver could teleport instantly — zero travel time — the chef still needs time to bake the pizza. Our calculation shows that 91% of the 272 ms wait is “baking time” (the AI generating the audio), not “delivery time” (data traveling across the network). Making the delivery faster only saves 9%. To get faster pizza, you need a faster oven or a pre-baked menu.

Technical Breakdown

  • Fiber speed: Light travels at ~200,000 km/s through glass fiber (about 2/3 of its speed in vacuum, slowed by the refractive index of the glass).
  • Round-trip to CDN: A 2,000 km path (typical US residential to a cloud CDN edge node) takes 20 ms round-trip — this is the physics minimum you cannot engineer away.
  • Persistent connection advantage: HTTP Persistent reuses the TCP and TLS connection. After the first request, those handshakes (normally 60–100 ms extra) are already paid. Only the 20 ms propagation remains per request.
  • Inference dominates: 272 ms − 20 ms (propagation) − 5 ms (HTTP framing) = 247 ms. This is the neural TTS model generating audio waveforms on Voice.ai’s servers.
  • Zero-network scenario: If Voice.ai were co-located with JARVIS (0 ms propagation), TTFB would drop to ~252 ms — only a 7% improvement over the current 272 ms.

  • Model optimization is the only big lever: Quantization, model distillation, or caching frequently spoken phrases would cut the 247 ms inference floor — network engineering cannot.
  • Edge inference nodes: Deploying inference servers closer to users (e.g., in Europe or Asia) saves ~30–50 ms per region — real but not transformative for US users.
  • “Under 300 ms” claims: From a co-located or same-region deployment, a sub-250 ms TTFB is achievable today. The current 272 ms is partly geography.

15.3   Hypothesis Testing: Mann-Whitney U Statistic

A non-parametric test requiring no normality assumption. Null hypothesis H₀: the two distributions are identical. Alternative H₁: Voice.ai TTFB is stochastically lower than Daniel TTFB. A t-test is inappropriate here because Daniel’s medium-sentence std is 4,006 ms — the data has heavy, non-normal tails.

HTTP Persistent short (n₁ = 8) vs. macOS Daniel short (n₂ = 13): Max(HTTP Persistent) = 397.1 ms Min(Daniel) = 1,428.4 ms → Perfect rank separation: 1,031 ms gap U₁ = n₁ × n₂ = 8 × 13 = 104 (all pairs won by Voice.ai) U₂ = 0 E[U] = n₁n₂ / 2 = 52 Var[U] = n₁n₂(n₁+n₂+1) / 12 = 8×13×22 / 12 = 190.67 SD[U] = √190.67 = 13.81 z = (U₁ − E[U]) / SD[U] = (104 − 52) / 13.81 = 3.76 p-value (two-tailed) ≈ 0.00017 (p < 0.001) Effect size (Cohen’s d): s_pooled = √[(7×44.6² + 12×1216.2²) / 19] = 967.0 ms d = |3151.7 − 291.6| / 967.0 = 2.96 (threshold for "huge": d > 0.8)
p ≈ 0.00017 — the result is not random. The probability of observing this rank separation by chance is 0.017%. Cohen’s d = 2.96 places the two systems in qualitatively different performance tiers — not just statistically different, but categorically incomparable.
How to understand the statistical significance test

In Plain English

Imagine 21 runners racing: 8 wearing Voice.ai jerseys and 13 wearing Daniel jerseys. When the race ends, every single Voice.ai runner (including the slowest, at 397 ms) crossed the finish line before the fastest Daniel runner (at 1,428 ms) even got there. That’s not just “Voice.ai usually wins” — it’s a complete sweep with a 1,031 ms gap between the two groups. The Mann-Whitney test asks: what are the chances this happened by accident? Answer: 0.017%. You’d have to run this race nearly 6,000 times before seeing this result randomly even once.

Technical Breakdown

  • Why not a t-test? The t-test assumes Normal distributions. Daniel’s medium-sentence data has a std of 4,006 ms, driven by a single 17,073 ms measurement — that’s a strongly non-normal tail. Mann-Whitney requires no distributional assumption and is the honest choice.
  • U statistic = 104: This is the count of (Voice.ai, Daniel) pairs where Voice.ai wins. With 8×13 = 104 possible pairs and Voice.ai winning all of them, U = 104 is the theoretical maximum — perfect separation.
