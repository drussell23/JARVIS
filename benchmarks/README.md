# Benchmarks — Trinity AI

Performance data collected automatically at the end of every full demo run.

---

## Files

| File | Description |
|------|-------------|
| `LATEST.md` | Most recent run in Markdown — always overwritten |
| `run-{timestamp}.json` | One file per run, timestamped in UTC, accumulates history |
| `BENCHMARK_RESULTS.md` | Historical hand-curated results and narrative |
| `benchmark_results.json` | Legacy benchmark data from earlier pipeline work |
| `ipc_and_inference_benchmark.py` | IPC and multi-tier inference benchmark script |

---

## How Benchmarks Are Generated

`LATEST.md` and `run-{timestamp}.json` are written by `demo_trinity_governed_loop.py` at the end of Phase 5.

To produce benchmark data, run the demo with J-Prime online:

```bash
python3 demo_trinity_governed_loop.py --no-voice          # silent run
python3 demo_trinity_governed_loop.py --no-voice --fast   # fast silent run
```

If J-Prime is unreachable, Phase 3 inference data will be missing and the benchmark files will not be written.

## How to Read the Numbers

### Latency

End-to-end time from HTTP request send to receiving the final `[DONE]` SSE token. Measured with `time.monotonic()` in the streaming thread. Includes network transit to GCP, model forward passes, and HTTP response framing.

**Typical range on g2-standard-4 + NVIDIA L4:**
- Short generation (100–130 tokens): 4,500–6,000 ms
- Medium generation (200–250 tokens): 9,000–11,000 ms

### Throughput (tok/s)

```
tok/s = completion_tokens / (latency_ms / 1000)
```

**Typical range:** 20–25 tok/s for Q4_K_M on L4.

This is *completion* throughput — the rate at which the model generates new tokens, not prefill throughput. Prefill (processing the system + user prompt) is much faster and is included in the latency figure but not in the tok/s calculation separately.

### Completion Tokens

Tokens generated by J-Prime (output only). Does not include prompt tokens. The demo uses `max_tokens` caps (250 and 200 respectively) to keep latency predictable for live presentation.

### Tests/Second

```
tests_per_second = passed / duration_s
```

Reflects pytest's parallelism and fixture overhead. Higher is faster, but the number reflects real test suite execution speed, not a synthetic benchmark.

## Quantization Note

All benchmarks in this directory use **Q4_K_M** quantization (GGUF standard 4-bit with k-quant mixed precision). This is the production quantization for `jarvis-prime`.

**Fisher Information optimal quantization (IQ2_M)** is a separate, more advanced system under development as the **Adaptive Quantization Engine**. It is designed to dynamically select the best quantization level per-layer based on Fisher information matrices, VRAM pressure, and task complexity. It is not active in the current demo. See:

- Spec: `docs/superpowers/specs/2026-03-14-adaptive-quantization-engine-design.md`
- Plan: `docs/superpowers/plans/2026-03-14-adaptive-quantization-engine.md`
- Implementation: `backend/core/memory_quantizer.py`

## Using Results in a Slide Deck

Key numbers to pull from `LATEST.md`:

| Metric | Where | Suggested framing |
|--------|-------|-------------------|
| **~24 tok/s** | Inference table, Throughput column | "Sub-second token latency on NVIDIA L4" |
| **~10s end-to-end** | Inference table, Latency column | "250-token secure infra code in 10 seconds" |
| **2,146 tests** | Governance Tests table, Passed row | "Full governance pipeline verified by 2,146 automated tests" |
| **99.6% pass rate** | Governance Tests table, Pass Rate row | "99.6% pass rate — 9 pre-existing structural failures, not pipeline regressions" |
| **Q4_K_M on L4** | System table | "Production quantization on dedicated GPU node" |

The 9 pre-existing test failures are documented failures in test harness infrastructure (`test_preflight.py`, `test_e2e.py`, `test_pipeline_deadline.py`, `test_phase2c_acceptance.py`) — they are not governance pipeline bugs and should be called out as such when presenting.

## Reproducing a Run

```bash
# Make sure J-Prime is online first
curl http://136.113.252.164:8000/health

# Run the demo
python3 demo_trinity_governed_loop.py --no-voice

# View the results
cat benchmarks/LATEST.md
```

Each run appends a new `run-{timestamp}.json`. Git-commit the `benchmarks/` directory after a run to preserve historical data:

```bash
git add benchmarks/LATEST.md benchmarks/run-*.json
git commit -m "chore: add benchmark run {timestamp}"
```
